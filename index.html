一、从 FILE * 到文件描述符

标准 I/O 使用 FILE *，系统调用 I/O 使用文件描述符 fd。文件描述符本质上是一个整型数，表示进程文件描述符表中的下标。

常见的三个标准文件描述符是：

  • 0：标准输入，stdin。
  • 1：标准输出，stdout。
  • 2：标准错误，stderr。

文件描述符进一步指向内核中的文件表项，文件表项再关联 inode。inode 保存文件属性，是文件在文件系统中的核心身份。

四、标准 I/O 与系统 I/O 的区别

  • 系统 I/O 响应速度快，但每次系统调用都要从用户态切换到内核态，有固定开销。
  • 标准 I/O 吞吐量大，通过缓冲批量处理，减少系统调用次数。
  • 两者都能用时，优先使用标准 I/O。
  • 标准 I/O 和系统 I/O 不建议混用，因为缓冲刷新时机不一致。

五、I/O 效率问题

修改 BUFFERSIZE，观察文件复制程序运行时间，找到效率较高的缓冲区大小。

这个实验很重要，因为系统调用不是越多越好，缓冲区也不是越大越好。缓冲区过小会导致系统调用频繁，过大可能增加内存压力，甚至影响缓存命中。实际项目中常见取值是 4KB、8KB、16KB、64KB，具体要结合文件系统、设备和业务场景测试。

七、文件系统：目录、链接与权限

重点包括目录和文件、系统数据文件和信息。

需要掌握的方向：

  • stat() / lstat()：获取文件属性。
  • inode：文件元数据的核心。
  • 硬链接：多个目录项指向同一个 inode。
  • 符号链接：一个特殊文件，内容是目标路径。
  • chmod() / chown()：权限和所有者。
  • umask()：影响新建文件默认权限。
  • opendir() / readdir()：目录遍历。

理解 inode 后，很多现象会变清楚：删除文件本质是删除目录项；只要还有进程打开该文件，磁盘空间可能不会立刻释放。

八、系统数据文件和进程环境

“系统数据文件和信息”“进程环境”。这部分通常包括：

  • /etc/passwd、/etc/group 等系统数据库。
  • getpwuid()、getpwnam()、getgrgid() 等接口。
  • 环境变量表 environ。
  • getenv() / setenv() / putenv() / unsetenv()。
  • 程序启动过程中的 argc、argv、环境表。

系统编程里，进程不是只执行代码。它还带着命令行参数、环境变量、打开的文件、当前工作目录、权限身份、资源限制等上下文。

九、进程基本知识

本章还进入了进程基础。建议重点掌握：

  • fork()：创建子进程。
  • exec() 系列：替换进程映像。
  • wait() / waitpid()：回收子进程。
  • exit() / _exit()：进程退出。
  • 僵尸进程与孤儿进程。
  • 进程组、会话和终端控制。

fork() 后父子进程几乎拥有相同的用户态地址空间，但通过写时复制优化，不会立刻复制全部内存。后续如果子进程调用 exec()，它会装载一个全新的程序。

十、实践建议

学习这一章时，不要只背 API。推荐按下面的顺序练：

  1. 手动实现cat命令。
  2. 手动实现myShell，支持简单命令执行和管道。
  3. 用 stat() 对比普通文件、目录、软链接、硬链接。
  4. 写一个小程序打印当前进程的参数、环境变量和当前工作目录。
  5. 写一个 fork + exec + wait 的命令执行器雏形。

总结

标准 I/O 更像“带缓冲的高级接口”，系统调用 I/O 更接近内核能力本身。文件描述符、inode、文件偏移量、进程环境和 fork/exec/wait 是系统编程的骨架。把这些概念串起来，后续的并发、IPC、高级 I/O 和网络编程都会更容易理解。

第二章、文件消息摘要与Base64编码

发表于 2026-06-07 分类于 Linux系统编程

一、摘要和编码解决的不是同一类问题

学习文件处理时，很容易把“摘要”“编码”“加密”混在一起。它们都可能把一段数据变成另一段数据，但目标完全不同：

  • 摘要算法：把任意长度输入映射为固定长度输出，主要用于完整性校验、防篡改、签名。
  • Base64 编码：把二进制数据变成可打印文本，方便在文本协议中传输。
  • 加密算法：通过密钥保护数据内容，目标是机密性。

所以，Base64 不是加密。任何人拿到 Base64 字符串都可以解码还原原始数据。

二、OpenSSL 库

OpenSSL 的安装方式：

  • 数字签名和验签。
  • TLS/SSL 网络通信。

三、摘要算法：MD5、SHA1、SHA256

摘要算法的几个性质：

  • 任意长度输入生成固定长度输出。
  • 输入只要发生微小变化，输出通常会明显变化。
  • 单向，不可逆。
  • 可用于文件完整性校验、签名、密码保存等场景。

需要补充一点：摘要输出不是“一定唯一”。理论上，不同输入可能得到相同摘要，这叫碰撞。安全摘要算法的目标是让碰撞在实践上足够难构造。

MD5

MD5 输出通常表示为 32 位十六进制字符串，可用于判断文件是否完整、文件验证、文件查重，但不适合安全加密，因为存在碰撞风险。

今天的工程实践中，MD5 更适合用于非安全场景，例如快速校验下载文件是否损坏。不要把 MD5 当作密码存储方案。

SHA1

SHA1 输出通常表示为 40 位十六进制字符串，比 MD5 更安全，但也存在碰撞风险。

SHA1 已经不适合作为安全签名场景的首选。历史项目中可能还能见到它，但新项目应优先选择 SHA256 或更高强度算法。

SHA256

SHA256 输出通常表示为 64 位十六进制字符串，目前被认为更安全，适合加密和数字签名等安全相关应用。

更准确地说，SHA256 本身不是加密算法，而是摘要算法。它常用于数字签名、证书、完整性验证、密码派生流程中的基础组件。

四、密码保存：SHA + salt 的正确理解

“密码保存一般使用 sha256 + salt”。这里的 salt 是随机生成的字符串，目的是让同一个密码在不同用户之间得到不同的摘要结果，防止彩虹表攻击和批量撞库优化。

虽然比裸 SHA256 好，但仍然抵抗不了高性能离线爆破。更推荐使用专门的密码哈希算法：

  • bcrypt
  • scrypt
  • Argon2
  • PBKDF2

这些算法会主动增加计算成本，减慢攻击者的爆破速度。对于系统编程学习来说，可以先理解 salt 的意义：同一个密码加不同 salt 后摘要不同，可以防止彩虹表攻击和批量撞库优化。

五、Base64 编码：可逆但会膨胀

Base64 的构成：

  • 大写字母、小写字母、数字。
  • 两个符号：+ 和 /。
  • = 表示结尾填充。

Base64 的目标是把任意二进制内容转换为可打印文本，例如图片、音频、视频、PDF 等。转换后体积大约变为原来的 4/3。

为什么会变大？因为 Base64 每 3 个字节，也就是 24 bit，拆成 4 组 6 bit，再映射到 64 个可打印字符。原本 3 字节变成 4 字符，因此膨胀约 33%。

六、Base64 的典型使用场景

Base64 常见于：

  • 邮件附件编码。
  • HTTP Basic Auth。
  • JSON 中传输小型二进制数据。
  • 证书、密钥、图片等内容的文本化存储。

但它不适合滥用。大文件不建议直接塞进 JSON 的 Base64 字符串里，因为会增加体积、内存占用和解析成本。更合理的方式通常是文件上传、对象存储或分块传输。

  • 文件分片与摘要校验

第一章、标准IO

发表于 2026-06-07 分类于 Linux系统编程

一、 I/O的重要性

在 Linux/Unix 系统编程中，I/O 是最基础也最容易被低估的一部分。无论是普通文件、目录、设备、管道还是 socket，最终都会以“可读写对象”的形式参与程序运行。先把标准 I/O 和文件对象理解清楚，后面学习系统调用 I/O、进程间通信、网络通信时会顺很多。在Linux中有一句很经典的话：“一切皆文件”，所有我们在linux中进行的操作都是基于文件进行的，而对文件操作无非就是读写，涉及到读写就必须借助IO。所以说IO很重要。

重点可以概括为一句话：区分标准 I/O 与系统调用 I/O，掌握文件打开模式、缓冲机制、文件定位、目录操作和常见练习任务。

二、文件概念：狭义文件与广义文件

文件的分类：

  • 狭义：文件和文件夹。
  • 广义：普通文件 -、目录 d、块设备 b、字符设备 c、链接 l、管道 p、套接字 s。

这也是“一切皆文件”思想的入口。系统编程里说的文件，往往不是只指磁盘上的 .txt、.c，而是指内核暴露出来的一类统一接口。比如终端可以读写，管道可以读写，网络 socket 也可以读写。理解这一点后，read/write、select/poll/epoll 等机制才会显得自然。

三、标准 I/O：围绕 FILE * 展开

标准 I/O 来自 C 标准库，核心对象是 FILE *。“FILE * 贯穿始终”。调用 fopen() 成功后得到的不是内核文件描述符，而是 C 库维护的文件流对象。

常见函数包括：

  • fopen()：打开文件。
  • fclose()：关闭文件，并刷新缓冲区。
  • fread() / fwrite()：按二进制块读写。
  • fgetc() / fputc()：按字符读写。
  • fgets() / fputs()：按字符串读写。
  • fseek() / ftell() / rewind()：文件定位。
  • fprintf()、perror()、strerror()：格式化输出与错误处理。

错误处理时要注意：errno 是宏，对应各种错误原因。实际写程序时，系统调用或库函数失败后，优先用 perror("xxx") 输出上下文，调试效率更高。

四、文件打开模式

常用 mode：

  • r：只读，要求文件必须存在。
  • r+：读写，要求文件必须存在。
  • w：只写，有则清空，无则创建。
  • w+：读写，有则清空，无则创建。
  • a：追加写，无则创建。
  • a+：读和追加写，无则创建。

这里最容易踩坑的是 w 和 w+：只要打开成功，原文件内容就会被截断。如果程序本意是“在文件末尾继续写”，应该使用 a 或 a+。
如果使用’r’或’r+’打开一个不存在的文件，fopen()会返回 NULL，并设置 errno 为 ENOENT（No such file or directory）。因此在使用这些模式时，务必检查返回值，避免后续操作在空指针上崩溃。

  • （1）手动实现一个cp命令。(这段代码有一个很重要的细节：如果目标文件打开失败，要先关闭已经打开的源文件。系统编程中资源释放顺序非常重要，尤其是后续涉及文件锁、进程、线程时。)
  • （2）手动实现一个 getline()，理解动态扩容。
  • （3）手动实现 wc 命令，统计行数、单词数、字节数。
  • （4）统计 C 源文件总行数、空行数、注释行数和代码行数。
  • （5）使用 fseek() / ftell() 切分文件。
  • （6）基于 DIR 实现一个简化版 ls。
  • （7）统计Linux内核源码目录中文件数、目录数以及所有 .c/.h 文件总行数。

总结

标准 I/O 的价值在于简单、可移植、吞吐量高。它通过缓冲减少系统调用次数，适合大多数普通文件读写场景。真正需要精细控制文件描述符、阻塞行为、文件锁、I/O 复用时，再进入系统调用 I/O。学完这一章，最重要的是把 FILE *、缓冲、文件定位和资源释放这几个点练熟。
